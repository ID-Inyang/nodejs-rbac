### 1. Types of Node.js / JavaScript Exports

In Node.js and modern JavaScript, there are two primary ways to export code from one file so it can be used in another:

* **CommonJS (Module.exports):** This is what Dipesh uses in the video (e.g., `module.exports = DBConnect` or `module.exports = { register, login }`). It is the default, traditional system in Node.js. You bring these files in using `require()`.
* **ES Modules (ESM):** This is the modern JavaScript standard that uses `export default function()` or named `export const myFunc`. You bring these in using `import` statements. To use this in Node.js, you usually have to add `"type": "module"` to your `package.json`.

---

### 2. Why do we even need a token on a successful login?

Think of a token like a digital wristband you get at a music festival.

When you first arrive at the festival gate, you show your actual ID and ticket (your **username and password**). The guard verifies it's you, but instead of making you pull out your ID and ticket every single time you want to buy a drink or enter a stage, they give you a secure **wristband (the JWT token)**.

HTTP is "stateless," meaning every single request your browser sends to the server is treated like a total stranger. Without a token, you would have to type your password again every time you clicked a new link on a website. The token is sent automatically with subsequent requests so the server knows, *"Ah, this request belongs to an already verified session."*

---

### 3. What's up with the "salt number" when hashing?

If you hash the password `"123456"` using a standard algorithm, it will *always* result in the exact same long string of random-looking characters.

If a hacker steals a database full of these standard hashes, they can use a "Rainbow Table" (a pre-computed dictionary of common passwords and their corresponding hashes) to instantly reverse-engineer everyone's passwords.

A **Salt** is a piece of completely random data added to the password *before* it gets hashed. The "salt number" (or salt rounds, which Dipesh sets to `10`) determines how many times the hashing algorithm scrambles the data.

* Because the salt is unique for every single user, two users with the exact same password (`"123456"`) will end up with completely different looking hashes in your database.
* It protects against Rainbow Table attacks and makes brute-forcing computationally exhausting for hackers.

---

### 4. How does Role-Based Access Control (RBAC) work with just a token?

You noticed something critical: the token is *generated* using the same code logic for everyone. However, the **payload (the data hidden inside the token)** is entirely custom for each user.

When Dipesh creates the token, he writes:

```javascript
const token = jwt.sign(
  { id: user._id, role: user.role }, // <-- The payload is unique!
  process.env.JWT_SECRET, 
  { expiresIn: "1h" }
);

```

When an Admin logs in, their token says `role: "admin"` inside. When a basic user logs in, their token says `role: "user"` inside.

Because the token is cryptographically signed using your secret key (`process.env.JWT_SECRET`), users cannot tamper with this data. If a regular user tries to open up their token and change `"user"` to `"admin"`, the signature will break, and the server's `verifyToken` middleware will reject it instantly.

### 5. What does "Protecting a Route" mean, and is it *real* security?

* **What it means:** A route is unprotected if anyone on the internet can type the URL into their browser or Postman and see the data. Making it "protected" means we insert a bouncer (a **middleware** function) in front of the route. The route handler will refuse to execute unless a valid token is produced first.
* **Is it real security?** Yes, this is industry-standard web security. When implemented over an encrypted connection (HTTPS), JSON Web Tokens are incredibly secure because they rely on robust mathematical cryptography.

---

### 6. Do you have to manually input the token in the header for every user?

**In development (testing with Thunder Client/Postman):** Yes, you manually copy and paste the token into the headers to simulate what a real application would do.

**In production (a real React, Vue, or mobile frontend):** No, the code does this automatically!

1. When a user logs in, your frontend code receives the token from the backend response.
2. The frontend saves that token securely (usually in browser `localStorage` or a secure cookie).
3. Using an HTTP client like **Axios**, developers set up an "Interceptor". This is a script that automatically grabs the stored token and attaches it to the `Authorization: Bearer <token>` header of *every single outgoing request* your app makes.

---

### 7. Resolving the confusion: "If I have a valid token, I can access all routes."

You pointed out a moment in the video where a Manager token was able to access the Admin route. This happened because of a classic structural flow in development:

Initially, Dipesh *only* attached the authentication middleware (`verifyToken`) to all the routes. At that specific timestamp, the server was only checking: *"Is this a real token signed by me?"* It was **not** checking *who* the token belonged to. Because the Manager had a validly signed token, the bouncer let them in.

To fix this, he immediately introduces the **Role Authorization Middleware** (`authorizedRoles`).

```javascript
router.get("/admin", verifyToken, authorizedRoles("admin"), ...);

```

Once that second layer is added, the process splits into two steps:

1. **Authentication (`verifyToken`):** "Is your wristband real?" $\rightarrow$ *Yes.*
2. **Authorization (`authorizedRoles("admin")`):** "Does your wristband say 'Admin' on it?" $\rightarrow$ *No, it says 'Manager'. Access Denied.*
